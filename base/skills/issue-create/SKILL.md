# Issue 作成 (GitHub)

GitHub の `gh` CLI を前提に、後から着手しやすい **構造化された issue** を起票する
ための手順。曖昧な一文 issue ではなく、背景・受け入れ条件・スコープが明示された
issue を作ることを目的とする。

> 関連: 起票した issue に着手して実装〜PR まで進めるときは `issue-implement` skill を使う。

## 適用する場面

- 新しい機能要望・バグ・改善・技術的負債を issue として残したいとき
- 口頭やチャットで出た要望を、追跡可能な issue に落とし込みたいとき
- 着手前に要件・受け入れ条件を整理しておきたいとき

## 手順

### 1. 事前確認

- 対象リポジトリを確認する (`gh repo view --json nameWithOwner -q .nameWithOwner`)。
- 重複がないか既存 issue を検索する。

  ```bash
  gh issue list --search "<キーワード>" --state all
  ```

- 既存 issue があれば、新規作成ではなくコメント追記や再オープンを提案する。

### 2. タイトル

- 簡潔に、要約形で書く。何をするか / 何が問題かが一目で分かること。
- 種別が分かるよう接頭辞を付けてもよい (例: `feat:` / `fix:` / `chore:` / `docs:`)。
- 良い例: `feat: ログイン画面にパスワード再設定リンクを追加`
- 悪い例: `バグ` / `お願い` / `ログインまわり`

### 3. 本文テンプレ

種別に応じて以下を埋める。不要なセクションは省く。空欄の見出しだけ残さない。

```markdown
## 背景 / 目的
なぜこの issue が必要か。解決したい課題やユーザー価値を 1〜3 文で。

## 受け入れ条件
- [ ] 満たすべき条件を検証可能な粒度で列挙する
- [ ] 「完了」と言える状態を明確にする

## スコープ外 (任意)
- 今回はやらないこと。誤解を防ぐために明示する。

## 再現手順 (バグのみ)
1. ...
2. ...
- 期待する挙動 / 実際の挙動

## 参考 (任意)
- 関連 issue / PR / ドキュメント / 設計メモへのリンク
```

### 4. メタ情報

- ラベル: 利用可能なラベルを確認して付与する (`gh label list`)。
- 担当・マイルストーンが分かっていれば指定する (`--assignee` / `--milestone`)。

### 5. 作成

- **issue 作成は外部公開アクション**。実行前にタイトルと本文をユーザーに提示し、
  内容の確認を取ってから作成する。
- 本文は長くなるためファイル経由 (`--body-file`) かヒアドキュメントを使うと安全。

  ```bash
  gh issue create \
    --title "feat: ログイン画面にパスワード再設定リンクを追加" \
    --body-file /tmp/issue-body.md \
    --label "enhancement"
  ```

- 作成後、返ってきた issue URL / 番号をユーザーに伝える。

## 注意

- issue 本文・タイトルに **シークレット・認証情報・個人情報 (PII)** を含めない。
  ログやスタックトレースを貼るときは秘匿情報をマスクする。
- 推測で受け入れ条件を埋めない。不明点はユーザーに確認するか「要確認」と明記する。
